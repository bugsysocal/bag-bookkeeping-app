## Flagged for review (schema / money-logic / auth-permission changes — batch review, not blocking)
Per user instruction (2026-07-18, extended 2026-07-19 to cover capability-gating decisions): changes that touch the schema, a migration, how money is calculated/moved, or who's allowed to do what, don't pause work, but land here with a one-line why, for batch review rather than real-time sign-off.
- **Migration `0004_compliance_ack.sql`** — added `vat_threshold_acked_fy_start`/`cit_threshold_acked_fy_start` (nullable TEXT) to `companies`. Why: Spec 07's Compliance banner is "non-editable [by owner], advisor-clearable only" — clearing without changing the underlying `vat_exempt`/`cit_exempt` flag needs somewhere to record "reviewed, no change needed for now," and nothing existed. Storing the fiscal-year-start the ack applies to (not just a timestamp) means a new fiscal year always re-surfaces the check, matching the project's existing "reminders follow the fiscal year" convention.
- **New Advisor-gated capability surface**: `post_manual_journal`, `update_tax_settings_cmd`, `update_hard_close_cmd`, `update_writeoff_settings_cmd`, `ack_vat_threshold_cmd`, `ack_cit_threshold_cmd`, and the `company_settings` read — all six gated with `require_advisor_elevated` (same guard as Trial Balance/GL), none previously had any command surface at all. `compliance_banners_cmd` (read-only) is deliberately **not** gated — same stance as `dashboard` itself, read-only data reachable without elevation.
- **`ledger-core` now depends on `serde`/`serde_json`** (previously avoided in favor of small hand-rolled codecs, e.g. the bank mapping-profile JSON). Why: the backup manifest is a growing array of records — a hand-rolled parser for that shape risks subtle bugs in a place where "the manifest looked fine but was quietly wrong" is a bad failure mode, and both crates are already proven to build fine here (unlike chrono) and already sit in the workspace's dependency tree via `tauri`/`calamine`. Not a schema/money/auth change, noted here anyway since it reverses an explicit earlier choice.
- **Compliance threshold check is revenue-only for the CIT leg.** The statutory small-company test is turnover ≤ ₦100M **and** fixed assets ≤ ₦250M, but there's no fixed-assets ledger in this app to check the second leg against — documented in `compliance.rs` rather than silently assumed. An advisor still has to apply judgment on the assets leg before touching `cit_exempt`.
- **Spec 07 §5 capability table — "Write-off above `writeoff_limit_kobo`" row corrected in the spec doc itself** (specs/07-dashboard-and-modes-ux.md), not just PROGRESS.md. Why: that row was written before Spec 04 §9 Decision #11 (closed 2026-07-07) removed the bypass entirely — the table said an Advisor-Mode capability existed that no longer does. Marked superseded in place, with the corrected rule and a pointer to Decision #11, rather than silently deleting the row (so the history of what changed and why stays visible in the spec itself, matching how Decision #11 was recorded in Spec 04).
- **WHT exemption override — left as ordinary (any-role) session access, not gated to `require_advisor_elevated`.** Spec 07 §5's capability table lists "WHT exemption override (both directions)" as Advisor-only, but the only WHT-override path that actually exists in code — `record_payment_out`'s `wht_mode`, exercised when a supplier has no TIN and Spec 01 §6.2 requires an explicit decision rather than a silent default — is reachable by any logged-in session today. Decision: keep it that way. Why: this is routine payment processing (Nigerian SMEs pay suppliers without a TIN on file constantly), and a payment can't correctly stall on advisor availability the way a genuine policy override could. My read is that the capability table means something not yet built — e.g. a settings screen to permanently mark a supplier WHT-exempt, or to override the `cit_exempt` gating judgment itself — not this per-payment decision. Flagged rather than silently resolved because the table's wording doesn't disambiguate, and a different reading would mean staff currently has an access they shouldn't.
- **Most of the Spec 07 §5 capability table has no UI/IPC surface to gate at all, yet** — informational, not a decision, but material to how "done" Spec 07's capability consolidation actually is: manual journal posting (`engine::post_journal` exists, no Tauri command), hard period lock, tax-flag/VAT-rate/WHT-preset editing, inventory on/off, `fiscal_year_start_month` change, write-off routing settings, OBE reclassification, opening-balance void & repost, bulk transaction import, and compliance-banner acknowledgment are all either engine-only or wizard-time-only with no post-setup edit path. Building all of these is out of scope for this pass (dashboard + Advisor Mode UX polish) — noted as follow-up, not attempted here.
- **Migration `0003_imported_paid.sql`** — added `imported_paid_kobo` (additive, default 0) to `invoices` and `bills`; `recompute_invoice_status`/`recompute_bill_status` now add it into the computed `paid` total. Why: without it, the first ordinary payment against an imported historical invoice/bill would silently erase the pre-migration collected amount, because that historical payment has no `payment_allocations` row to back it (caught by a failing test before commit, not by a user report).
- **`engine::import_open_invoice` signature change** — now takes a caller-supplied `invoice_no` and stores it directly as the invoice's `number`, instead of drawing the next number from `document_sequences`. Why: Spec 06 §3 lists "invoice no.*" as a required import field and "company + invoice no." as the dedup key — the prior system's own numbering must carry over, not be replaced. Note: there is no DB-level uniqueness constraint on `(company_id, number)`, so a future in-app-generated invoice number colliding with an imported historical one is a latent (very low-probability, unenforced) risk worth a look.
- **Open Bills importer includes an "original total" column** not present in the Spec 06 §3 table for bills (which lists only "balance outstanding*"). Why: `engine::import_open_bill` needs both to validate a partial balance against the original amount, exactly as it does for invoices, and §3.1's own text assumes an original-total field exists for display. Treated the spec table as incomplete rather than skip validation.
- **WHT rate cell parsing reuses `csv_util::parse_kobo_cell`** (naira→kobo, ×100) to turn a percentage cell into basis points (percent × 100 = bp — the same scaling operation) instead of a dedicated percentage parser. Mathematically correct, but it's a repurposing of a money-parsing helper for a non-money rate field, worth a human eyeball.

## For a new developer joining this project

You are inheriting a **spec-first** codebase: every feature was written as a reviewed markdown spec in `specs/` **before** any code. When in doubt about intended behaviour, the spec is the source of truth, and [specs/00-key-decisions-digest.md](specs/00-key-decisions-digest.md) is the one-page map to all of it. This file (PROGRESS.md) is the authoritative *status* doc — what is built, what is not, what to do next.

### Build & run from a clean clone
Prerequisites: **Rust** (stable, MSVC toolchain), **Node.js** (only needed if you regenerate Tauri assets; the committed `ui/` is plain HTML/JS and needs no build step). No database setup — SQLite is embedded (`rusqlite` `bundled` feature) and the schema is created on first run by the migration runner in `crates/ledger-core/src/db.rs`.

```
cargo test --workspace          # run the whole suite (currently 104 tests)
cargo build -p ledgerone-desktop   # build the desktop app
cargo run   -p ledgerone-desktop   # launch it (creates its DB under %APPDATA%\ng.boltonadvisory.ledgerone)
```
The running app's data lives at `%APPDATA%\ng.boltonadvisory.ledgerone\ledgerone.db`. To reset to a first-run state, rename/move that file (do not delete blindly — it is real books once used).

### Windows / ARM64 toolchain gotchas (this dev machine is Windows on ARM64)
These bit the previous developer; they will bite you too on a similar machine:
- **`VC.Tools.ARM64` MSVC component**: the VS 2022 "VCTools" workload installs x64-host tools by default. On an ARM64 host you must add the ARM64 component separately or linking fails with "link.exe not found". Install via winget with `--force --override "... --add Microsoft.VisualStudio.Component.VC.Tools.ARM64"`; the VS installer's own `setup.exe --quiet` is blocked from a non-elevated shell (exit 5007) — go through winget.
- **rustup `settings.toml` conflict**: if a stale `~/.rustup/settings.toml` points at a toolchain that no longer exists, `cargo` errors before it does anything. Fix by reinstalling the stable toolchain (`rustup toolchain install stable-aarch64-pc-windows-msvc`) or clearing the stale default.
- **`cargo` on PATH**: lives at `%USERPROFILE%\.cargo\bin`. Pre-existing shells may not have it until PATH is refreshed; call it by full path if `cargo` is "not recognized".
- **Windows Defender + `target/`**: Defender scans freshly-linked binaries and intermittently holds a lock, causing `LNK1104`/`LNK1201`. **Add the project `target/` folder to Defender exclusions** (needs admin) — this is the durable fix. Stop-gap without admin: `Remove-Item target\debug\deps\*.exe, target\debug\deps\*.pdb` then rebuild.
- **`chrono` is banned from `ledger-core`**: its `num-traits`/`autocfg` build script fails in this environment. Timestamps are hand-rolled std-only civil-date math in `ids.rs` — do not reintroduce chrono.

### Architectural boundary — do not cross it
This is the single most important rule in the codebase:

> **All posting/business logic lives in `crates/ledger-core`. The Tauri shell (`src-tauri/src/lib.rs`) contains only thin command wrappers that (de)serialize DTOs and call `ledger-core`. Never write ledger logic, SQL that posts, or accounting rules in the shell.**

- Every financial event flows through one posting surface (`engine.rs` functions → `posting::post_entry_in`) inside one DB transaction; the balance guarantee is enforced by SQLite triggers (Spec 01 §4), not by app code. See [specs/00-key-decisions-digest.md](specs/00-key-decisions-digest.md) "five invariants."
- Errors returned to the UI use the `CmdError` envelope: an owner-language message + a next step, with any technical detail carried separately for Advisor Mode. Every new command must keep this — no raw engine strings reach the owner.
- The frontend (`ui/index.html`) is a single dependency-free HTML/JS file talking to commands via `window.__TAURI__.core.invoke`. Its only capabilities are core IPC + the opener plugin (wa.me/mailto). No shell-exec, no fs, no arbitrary network.
